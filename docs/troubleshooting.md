# troubleshooting.md

## 1) 재고 동시성 문제

### 문제 상황
- 동일 제품에 대해 여러 작업자가 거의 동시에 생산 실적 입력
- 재고 업데이트가 경합하면서 최종 수량이 실제보다 작거나 크게 반영되는 이슈 발생

### 해결 방법
- 재고 조회 시 잠금(`SELECT ... FOR UPDATE` 또는 MSSQL 락 힌트) 적용
- 서비스 계층에서 재고 반영 + 로그 저장을 단일 트랜잭션으로 묶음
- 필요 시 낙관적 락(`version` 컬럼) 도입해 충돌 감지 후 재시도

### 배운 점
- “재고”는 정합성 우선 도메인이라 성능보다 잠금/일관성 전략이 먼저 설계되어야 함
- 동시성 테스트(멀티스레드/부하테스트)는 개발 초기부터 포함해야 함

## 2) 트랜잭션 처리 문제

### 문제 상황
- 생산 실적은 저장됐는데 재고 로그는 누락되는 부분 성공(Partial Success) 사례 발생 가능
- API 예외 처리 방식에 따라 롤백이 누락될 위험 존재

### 해결 방법
- `@Transactional` 범위를 생산실적 저장 로직 전체로 확장
- Checked/Runtime 예외 정책 정리하여 롤백 대상 명확화
- 외부 연동 없이 DB 처리만 포함해 트랜잭션 경계를 단순화

### 배운 점
- 트랜잭션 경계는 “기능 단위”가 아니라 “정합성 단위”로 잡아야 함
- 실패 시 어떤 데이터까지 롤백되어야 하는지 먼저 정의해야 실수가 줄어듦

## 3) 상태 관리 설계 고민

### 문제 상황
- 상태 전이 규칙이 명확하지 않으면
  - 완료된 작업지시에 실적 추가
  - 음수/과다 입력 등 비정상 케이스 발생
- 상태 값이 문자열로 분산 관리되면 오타/누락 위험 증가

### 해결 방법
- 상태 Enum(`PLANNED`, `IN_PROGRESS`, `COMPLETED`)으로 타입 안전성 확보
- 상태 전이 로직을 WorkOrder 도메인 메서드에 집중
- API 단에서 상태별 허용 동작을 명시적으로 검증

### 배운 점
- 상태는 단순 컬럼이 아니라 비즈니스 규칙의 핵심
- “전이 가능/불가능”을 표로 먼저 정의하고 코드로 옮기면 유지보수가 쉬워짐

## 면접에서 강조할 포인트

1. **동시성 제어 전략**: 왜 락을 선택했고 어떤 부작용을 감수했는지
2. **트랜잭션 설계**: 어떤 단위를 원자적으로 묶었는지
3. **상태 모델링**: 상태 전이 규칙을 코드에 어떻게 강제했는지
4. **로그 추적성**: 장애/이슈 발생 시 원인 역추적이 가능한지
